The core idea · Leverage
Small thing builds big thing
A language model can only hold so much in its head at once. Ask it to build a whole word processor in one go and it drowns. The trick is composition: break the large problem into small problems the model can actually handle — then build a machine that runs that decomposition, one small problem at a time, forever.

And here's the recursion: the problem of doing that composition well is itself one of those big problems. So we composed it too — that's what the foundry is.
Act I · The pattern
A dev machine is one loop
Strip away the tooling and it's a single disposable cycle. Every session starts from zero context, reads the spec before it writes code, and verifies before it commits.
read STATE.yaml → pick next feature → spawn zero-context session → implement from spec (TDD) → verify build + test → commit → update state → repeat
Every session is disposable
It reads state, does 3–5 features, writes state, and exits. Nothing lives in a session's memory.
The spec is the product
"If your specs are good, the machine builds good software. If your specs are vague, it builds vague software."

Pattern · Progressive design
Three layers, gated
Layer 1
The Constitution
One architecture spec. Written once, reviewed hard, rarely changed. Prevents drift across hundreds of features.
Layer 2
Module specs
One per module. Written just before that module is built.
Layer 3
Feature specs
Lightweight work orders, in batches: interfaces, acceptance criteria, edge cases, files to touch.
⛓️ The gating rule: Layer N+1 cannot be written until Layer N is approved.
word4's constitution was a 947-line spec written in about an hour — then it evolved. "Sufficient" means enough to prevent drift, not enough to anticipate everything.
Pattern · State management
Truth lives in files
No session remembers anything. So the machine writes everything down: the current phase, every work item with its status and commit hash, and the blockers list — whose mere presence halts the machine.
STATE.yaml
Machine-readable truth. Updated after every feature, never batched. Carries a next_action field telling the next session exactly what to do.
Built-in quality cadences
Smoke tests every 5 sessions · performance profile every 15 · state pruning every 20. The machine audits itself on a clock.
The cardinal rule: never two work items without a state update between them. One feature at a time through the state machine.
Pattern · Testing
Tests first — then a fresh agent tries to break it
Tier 1 · Unit
Seconds. After every feature. TDD: RED before GREEN.
Tier 2 · Visual QA
SSIM structural-similarity scoring against reference renders. Pass ≥ 0.85.
Tier 3 · Live E2E
Against the live staging deploy, gated to run every ≥ 5 commits.
Antagonistic review
After each feature, a zero-context sub-agent reads the spec and the diff and actively hunts for failures. In one session it caught a bug where fontSizeCs=0 would have made text invisible — a missing >0 guard.
It even has the spine to say no: the machine rejected feature F-1546 on technical merit after self-review found the mapping unsound — and added a negative test so it could never be reintroduced.
Pattern · CI/CD & recovery
Three layers that keep it alive
♻️
Entrypoint retry loop · inside container · continuous
Handles API blips, Cloudflare challenges, recipe crashes. Exponential backoff to a 45-minute ceiling.
🐕
Watchdog · host cron · every 15 min
Container crashes, OOM, host reboots. Checks heartbeat freshness before restarting.
📡
Monitor · host cron · every 10 min
Operational drift: stale blockers, unpushed commits — auto-fixed when safe.
Hard-won detail: the recipe CLI exits 0 even when a step fails. The entrypoint now scans stdout for "Recipe execution failed" — because once, a silent loop burned an entire night across 514 attempts.
Pattern · Context preservation & refactoring
How a memoryless machine remembers
CONTEXT-TRANSFER.md
Institutional memory. Reverse-chronological session summaries. In word4 it grew to 6,258 lines.
SCRATCH.md
Disposable working memory. The next session may ignore it — it belonged to the last one.
AGENTS.md
The guardrail every human session hits: "you run the machine, you are not the machine."
Refactoring is on a leash too
Any module past 10,000 LOC means STOP adding features and file a refactor plan. That rule split editor-core from 15,613 → 8,563 LOC — automatically, mid-run.
Act II · The foundry
Before it builds, it decides if it should
Not every problem is eligible for autonomous development. The foundry's /admissions process interrogates your project first — it reads your codebase, runs your commands, and scores your actual artifacts against a set of gates.
What it is
A confidence-scored readiness assessment that prescribes exactly what to fix when you're not ready.
What it is not
A rubber stamp. "It doesn't take your word for anything."
Scoring is brutally honest by design: below 50% is a hard stop, 50–75% is proceed-with-caution. "No rounding up. No optimism bias. A 48% is a 48%."
Act II · The foundry · Admissions
The gates
1Decomposability — hundreds of small, independent units?
2Verifiable correctness — fast, automatic feedback?
3Sufficient architecture — enough to prevent drift?
4Functioning toolchain — does build & test run?
5Spec quality — specs a machine can consume?
6Intent clarity — clear enough to work from alone?
A note on evolution: the README still says five gates. The implemented advisor has six — Intent Clarity was added later. The pattern is still learning about itself.
Real assessment
safeguard: 15%
on Gate 4 — zero code, no build system, no toolchain chosen. The assessment prescribed exactly what to do: pick the stack, scaffold, verify a clean build.
✅After remediation, the machine ran fine — and finished 100% healthy.

Act III · The honest part
The Ruler Incident
word4 didn't stop at 5.5 days. Left running, it kept shipping "features." And a basic one — a ruler, the thing every word processor has at the top — sat in the machine's next_action field as a candidate, session after session.
22
consecutive sessions it was carried forward — and never selected
0
lines of ruler code, after 534 sessions
The machine always chose another OOXML property to map instead. Deterministic, always available, always passes review, increments the counter. The ruler required design judgment — so it never got picked.
Act III · The honest part
Completionism, not value
By session 534 the numbers told the story. The machine had found a repeatable, always-available pattern — mapping OOXML properties through six layers — and optimized for it, because user-visible work is harder and less predictable.
4.06:1

The machine doesn't lie or hide issues — STATE.yaml faithfully reported every problem. It just can't prioritize. That's the human's job.
Act III · The honest part
Same pattern, different fates
Run across four projects, the pattern's outcomes spread out — and the failures were almost always in the gap between "template generated" and "machine fully operational," not in the architecture itself.
safeguard · the reference model
228 iterations · 100% healthy · blocker-free throughout
openM365 · critical failure loop
160 iterations · 1.25% · a build-gate + infinite retry trap
universal-app · stalled at Phase 8
built a platform, then hallucinated APIs when asked to code against it
The architecture is robust when properly configured. Most failures live in setup and configuration — the boring part the admissions gates exist to catch.
The takeaway
The machine is the leverage.
The judgment stays human.
Compose a problem too big for the model into small problems it can handle, and you can build things the model never could in one pass. The foundry is that same move applied to itself — small tooling that builds the machine that builds the software.
What the machine gives you
Tireless throughput, disciplined verification, honest state — hundreds of features, one spec at a time.
What it still needs from you
Priorities. The sense of what's worth building next. The ruler nobody told it to draw.
